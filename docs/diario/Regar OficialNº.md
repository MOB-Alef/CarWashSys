REGRA 001

Todos os Apps terão exatamente
a mesma estrutura.

---------------------------------

REGRA 002

Todo código importante será revisado
antes da próxima etapa.

---------------------------------

REGRA 003

Todo encerramento de dia terá um commit.

---------------------------------

REGRA 004

Todos os arquivos principais terão
comentários indicando o Dia em que
foram criados ou modificados.

---------------------------------

REGRA 005

Nenhuma funcionalidade será implementada
sem antes existir documentação.

---------------------------------

REGRA 006

Ao final de cada dia de desenvolvimento deverão ser atualizados, quando aplicável:

✅ docs/diario/DIA_X.md
✅ docs/diario/History_CWS.md
✅ CHANGELOG.md
✅ ROADMAP.md

A documentação deverá evoluir junto com o código.

---------------------------------

REGRA 007

Toda alteração de arquitetura deverá ser discutida antes de ser implementada.

O objetivo é preservar a organização, evitar retrabalho e manter o sistema preparado para crescer.

---------------------------------

REGRA 008

Sempre que uma melhoria for proposta, ela deverá responder positivamente às seguintes perguntas:

Facilita a manutenção daqui a alguns anos?
Torna o sistema mais organizado?
Será útil quando a empresa crescer?

Se a resposta for "sim", a melhoria poderá ser implementada.

Se a resposta for "não", ela deverá ser reavaliada ou adiada.

---------------------------------

REGRA 009

Nenhum módulo será considerado concluído apenas por funcionar.

Um módulo somente será considerado finalizado quando atender aos seguintes critérios:

Código organizado.
Documentação atualizada.
Estrutura padronizada.
Revisão concluída.
Testes executados sem erros.

---------------------------------

REGRA 010

O aprendizado faz parte do projeto.

Cada etapa deverá contribuir não apenas para o desenvolvimento do sistema, mas também para o crescimento técnico de quem o desenvolve.

O conhecimento adquirido durante o processo é tão importante quanto o software entregue.

---------------------------------

REGRA 011

Antes de escrever qualquer código novo, devemos perguntar:

"Se eu abrir este arquivo daqui a dois anos, conseguirei entendê-lo com facilidade?"

Se a resposta for não, então o código ainda pode ser melhorado.

Essa regra resume praticamente toda a filosofia do projeto: escrever código pensando no futuro, não apenas no presente.

---------------------------------
